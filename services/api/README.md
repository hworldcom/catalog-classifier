# Catalog Classifier API

This service owns the Hypertext Transfer Protocol (HTTP) application programming
interface (API), durable workflow metadata, and database migrations for the catalog
classifier.

The repository-level [`README.md`](../../README.md) remains the source of truth for the
overall product architecture and implementation sequence. This document records
service-level engineering decisions and operating instructions.

## Current Boundaries

- The existing `/v1/upload-handshake` and `/v1/local-batches` proof-of-concept routes
  continue to use temporary or local filesystem state.
- PostgreSQL stores metadata and workflow state only.
- Original images, thumbnails, and other file contents must remain outside PostgreSQL.
- Classification, grouping, review history, authentication, and cloud object storage
  integration are not part of the initial database migration.

## Google Cloud Upload Setup

The upload foundation work in ticket `0006` depends on the following manual Google
Cloud setup in the development project:

- a private Cloud Storage bucket for ingestion files;
- bucket CORS allowing origins `http://localhost:3000` and
  `http://127.0.0.1:3000`, method `PUT`, and request header `Content-Type` so the
  local browser frontend can send direct signed upload requests;
- a dedicated API service account with `roles/storage.objectCreator` on that bucket;
- the same API service account also needs `roles/storage.objectViewer` on that
  bucket so finalization can inspect object metadata;
- signed upload URLs generated by the API, with file bytes uploaded directly from the
  browser to Cloud Storage.
- the backend reads the bucket name from `CATALOG_UPLOAD_BUCKET`.
- the project has the Service Account Credentials API enabled.
- the signing principal has `iam.serviceAccounts.signBlob`, typically through
  `roles/iam.serviceAccountTokenCreator`, on the service account that signs URLs.

The current development values are:

- bucket: `gs://lnlabs-bucket`;
- API service account: `catalog-api@catalog-classifier.iam.gserviceaccount.com`.

Grant the bucket-level upload and metadata permissions:

```bash
gcloud storage buckets add-iam-policy-binding gs://lnlabs-bucket \
  --member="serviceAccount:catalog-api@catalog-classifier.iam.gserviceaccount.com" \
  --role="roles/storage.objectCreator"

gcloud storage buckets add-iam-policy-binding gs://lnlabs-bucket \
  --member="serviceAccount:catalog-api@catalog-classifier.iam.gserviceaccount.com" \
  --role="roles/storage.objectViewer"
```

Verify the policy and effective object-metadata access:

```bash
gcloud storage buckets get-iam-policy gs://lnlabs-bucket

gcloud storage objects describe "gs://lnlabs-bucket/<object-key>" \
  --impersonate-service-account=\
catalog-api@catalog-classifier.iam.gserviceaccount.com
```

This setup keeps the bucket private while allowing the browser to upload files
without proxying them through the API service.

Local development uses Application Default Credentials with service-account
impersonation, so no downloaded key files are needed. The upload URLs are V4 signed
`PUT` URLs with a 15-minute expiration. The signer should be isolated behind an
interface so tests can replace it with a fake implementation.

Configure local credentials:

```bash
gcloud auth application-default login \
  --impersonate-service-account=catalog-api@catalog-classifier.iam.gserviceaccount.com
gcloud config set project catalog-classifier
export CATALOG_UPLOAD_BUCKET=lnlabs-bucket
export CATALOG_SIGNING_SERVICE_ACCOUNT="catalog-api@catalog-classifier.iam.gserviceaccount.com"
```

The signing service account variable defaults to the development account shown above,
but setting it explicitly makes the active identity clear.

## Database Foundation

PostgreSQL 16 is the supported local database. SQLAlchemy 2 defines the application
models and Alembic owns schema migration history.

### Type decisions

- Identifiers use PostgreSQL `uuid` values generated by the database with
  `gen_random_uuid()`.
- Timestamps use timezone-aware `timestamp with time zone` values and database-generated
  creation times.
- File sizes use `bigint`; counters, dimensions, and upload order use `integer`.
- Object keys, filenames, and error messages use `text`.
- Hashes and lifecycle states use bounded character columns.
- Lifecycle values are strings protected by database `CHECK` constraints rather than
  PostgreSQL enum types. This keeps future status migrations explicit and reversible.

### `organizations`

| Column | Type | Null | Default |
| --- | --- | --- | --- |
| `id` | `uuid` | no | `gen_random_uuid()` |
| `name` | `text` | no | none |
| `created_at` | timezone-aware timestamp | no | `now()` |

Deleting an organization is restricted while upload batches or image assets reference it.

### `upload_batches`

| Column | Type | Null | Default |
| --- | --- | --- | --- |
| `id` | `uuid` | no | `gen_random_uuid()` |
| `organization_id` | `uuid` | no | none |
| `created_by` | `uuid` | yes | none |
| `status` | `varchar(32)` | no | `created` |
| `original_file_count` | `integer` | no | `0` |
| `processed_file_count` | `integer` | no | `0` |
| `pipeline_version` | `varchar(100)` | yes | none |
| `created_at` | timezone-aware timestamp | no | `now()` |
| `finalized_at` | timezone-aware timestamp | yes | none |
| `completed_at` | timezone-aware timestamp | yes | none |

Allowed statuses are `created`, `uploading`, `queued`, `processing`,
`review_required`, `approved`, `failed`, and `cancelled`.

Counts cannot be negative, and `processed_file_count` cannot exceed
`original_file_count`. The pair `(id, organization_id)` is unique so image assets can
reference a batch without crossing organization boundaries.

### `image_assets`

| Column | Type | Null | Default |
| --- | --- | --- | --- |
| `id` | `uuid` | no | `gen_random_uuid()` |
| `organization_id` | `uuid` | no | none |
| `batch_id` | `uuid` | no | none |
| `original_object_key` | `text` | no | none |
| `thumbnail_object_key` | `text` | yes | none |
| `original_filename` | `text` | no | none |
| `upload_order` | `integer` | no | none |
| `mime_type` | `varchar(255)` | no | none |
| `size_bytes` | `bigint` | no | none |
| `width` | `integer` | yes | none |
| `height` | `integer` | yes | none |
| `sha256` | `varchar(64)` | yes | none |
| `status` | `varchar(32)` | no | `pending` |
| `error_code` | `varchar(100)` | yes | none |
| `error_message` | `text` | yes | none |
| `created_at` | timezone-aware timestamp | no | `now()` |

Allowed statuses are `pending`, `uploaded`, `processing`, `processed`, and `failed`.
File size, upload order, and dimensions cannot be negative.

The schema enforces:

- one upload order per organization and batch;
- one original object key per organization;
- one non-null thumbnail object key per organization;
- a composite batch relationship on `(batch_id, organization_id)`;
- cascading image metadata deletion when a batch is deleted.

The ordered batch query is supported by the unique index backing
`(organization_id, batch_id, upload_order)`.

## Default Organization

The initial migration creates this local development organization idempotently:

```text
id:   00000000-0000-0000-0000-000000000001
name: Default Organization
```

The migration removes that row during downgrade after dependent tables have been
removed.

## Batch Creation Notes

- `POST /v1/upload-batches` reuses the existing `MAX_FILES_PER_REQUEST = 20` constant.
  No separate environment configuration exists for that limit yet.
- The API assumes Alembic migrations have already been applied before startup. It does
  not run migrations automatically.
- Request handlers should use request-scoped SQLAlchemy sessions.
- Unexpected database failures surface as `500 Internal Server Error` during this proof
  of concept stage.
- Tests verify new batches by querying through SQLAlchemy rather than by adding a
  separate batch retrieval endpoint.

The endpoint accepts no request body and returns:

```json
{
  "batchId": "uuid",
  "status": "created",
  "maxFiles": 20
}
```

Every successful request creates a separate row under the default organization. The
insert supplies only `organization_id`; PostgreSQL generates the identifier, status,
counts, nullable values, and creation timestamp from schema defaults.

If persistence fails, the transaction is rolled back and the API returns:

```json
{
  "detail": {
    "code": "database_error",
    "message": "Unable to create the upload batch."
  }
}
```

## File Registration

Register declared file metadata against a newly created batch:

```http
POST /v1/upload-batches/{batchId}/uploads
Content-Type: application/json
```

```json
{
  "files": [
    {
      "originalFilename": "front.jpg",
      "mimeType": "image/jpeg",
      "sizeBytes": 123456
    }
  ]
}
```

The request accepts one to 20 files. Filenames must be nonblank, the declared media
type must be `image/jpeg`, and sizes must be between 1 and 10,485,760 bytes. Duplicate
filenames are allowed. The array position becomes `upload_order`.

Successful registration:

- locks the batch row;
- requires the batch to still be `created`;
- creates all image rows in one transaction;
- changes the batch to `uploading`;
- sets `original_file_count` and leaves `processed_file_count` at zero;
- returns a Google Cloud Storage V4 signed `PUT` URL for every image.

The signed URL binds `Content-Type: image/jpeg` and expires after 15 minutes. This
endpoint trusts declared metadata only. Finalization will verify the uploaded object
bytes, type, and size.

Error behavior:

- malformed JSON fields or malformed batch identifiers return `422`;
- valid but unacceptable file metadata returns `400`;
- valid but unknown batch identifiers return `404`;
- registration against any state other than `created` returns `409`;
- database or signing failures return `500` and roll back the registration.

Registration is not idempotent. A later re-sign endpoint will refresh expired URLs for
existing image rows without creating replacements. Retryable rows may be assigned a
new object key during that flow so stale upload objects are not reused.

## Local Setup

Create the Python environment and install the service:

```bash
cd services/api
python3 -m venv .venv
.venv/bin/python -m pip install -e '.[dev]'
```

Start PostgreSQL from the repository root:

```bash
docker compose up -d postgres
```

The default connection string is:

```text
postgresql+psycopg://catalog:catalog@localhost:5432/catalog_classifier
```

Override it with `CATALOG_DATABASE_URL`. Run migrations from `services/api`:

```bash
.venv/bin/alembic upgrade head
.venv/bin/alembic current
.venv/bin/alembic downgrade base
```

Start the API after upgrading:

```bash
export CATALOG_UPLOAD_BUCKET=lnlabs-bucket
export CATALOG_SIGNING_SERVICE_ACCOUNT="catalog-api@catalog-classifier.iam.gserviceaccount.com"
.venv/bin/uvicorn catalog_api.main:app --reload --port 8000
```

Set these variables in the same terminal that starts Uvicorn. Restart the API after
adding or changing them because a running process does not inherit later shell changes.
If the bucket configuration or signing credentials are unavailable, file registration
returns `500` with error code `upload_registration_failed`. The registration
transaction is rolled back: the batch remains `created`, and no image rows are retained.

A successful registration response confirms that metadata and signed URLs were
created. It does not confirm that image bytes reached Cloud Storage. The client must
then send each file with the returned signed `PUT` URL.

## Batch Finalization and Readback

After all intended uploads complete, the backend verifies the registered Cloud
Storage objects through:

```http
POST /v1/upload-batches/{batchId}/finalize
GET  /v1/upload-batches/{batchId}
```

Finalization checks object existence, `image/jpeg` content type, and exact size.
Successful finalization marks image rows `uploaded`, sets `finalized_at`, and moves
the batch to `queued`. A `queued` batch is already verified and should not be checked
against Cloud Storage again. Partial verification failure keeps the batch `uploading`
and preserves per-image error details, while authentication, permission, network, or
provider failures return `500`. When the storage inspector creates impersonated
credentials, it requests a read-only Cloud Storage OAuth scope; bucket-level
`roles/storage.objectViewer` remains the authorization control.

## Upload Retry URLs

An `uploading` batch can receive fresh signed URLs for selected `pending` or `failed`
images through:

```http
POST /v1/upload-batches/{batchId}/retry-failed
```

The request contains unique image identifiers from the batch. The backend locks the
batch first, locks the selected images in `upload_order`, validates the complete
selection, and assigns each selected row a key with this form:

```text
organizations/{organizationId}/batches/{batchId}/originals/{imageId}/retries/{retryToken}.jpg
```

`retryToken` is a new UUID for every attempt. The new keys and signed `PUT` URLs are
created as one transaction; signing or database failure rolls back every key change.
The operation does not change image status, error fields, batch counts, or timestamps.
Earlier Cloud Storage objects remain outside the current database references and are
left for the later stale-object cleanup slice.

## Validation

Normal API tests do not require PostgreSQL:

```bash
.venv/bin/pytest
```

Migration checks create and remove isolated temporary databases. Run them against the
local PostgreSQL service:

```bash
CATALOG_TEST_DATABASE_URL=postgresql+psycopg://catalog:catalog@localhost:5432/postgres \
  .venv/bin/pytest
```

The PostgreSQL checks verify upgrade, schema and model alignment, constraints, ordered
image retrieval, the default organization seed, batch endpoint persistence and failure
handling, and downgrade with all application tables removed. Alembic retains its empty
`alembic_version` bookkeeping table.

Verify the real signing configuration without uploading an object:

```bash
CATALOG_UPLOAD_BUCKET=lnlabs-bucket \
CATALOG_SIGNING_SERVICE_ACCOUNT=\
catalog-api@catalog-classifier.iam.gserviceaccount.com \
  .venv/bin/python -c \
  "from catalog_api.upload_storage import get_upload_url_signer; \
print(get_upload_url_signer().sign_upload_url(\
object_key='qa/signed-url-check.jpg', content_type='image/jpeg'))"
```
